# Learning Guide

## Working rule

Every study session must produce something that can be committed: code, a test, documentation, a lab result, or a verified engineering note.

## Session format

1. Define one observable outcome.
2. Study only the concepts required for that outcome.
3. Build a small working change.
4. Run validation.
5. Explain the result in your own words.
6. Record the session.
7. Commit the work.

## Recommended session length

- 15 minutes: review and objective
- 30–45 minutes: guided learning
- 45–90 minutes: implementation
- 15 minutes: testing and engineering log

## How to use AI

Use AI to:

- explain errors;
- compare approaches;
- generate test cases;
- review code;
- identify missing assumptions;
- suggest documentation improvements.

Do not accept code that you cannot explain. Before committing generated code, answer:

- What problem does it solve?
- What are its inputs and outputs?
- How can it fail?
- How is it tested?
- Why does it belong in this module?

## Definition of done

A task is done only when:

- the acceptance criteria are met;
- relevant checks pass;
- no credentials or generated junk are committed;
- documentation matches behavior;
- the engineering log is updated when meaningful;
- the commit message describes the change.
